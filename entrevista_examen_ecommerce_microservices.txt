BLOQUE 1 — VISIÓN GENERAL Y DECISIONES DE STACK
================================================================================

P1. Describe la app en términos funcionales y tecnológicos.

R1. Es un mini e-commerce backend con microservicios. Exponemos un único
    punto de entrada con API Gateway (`api-gateway`) que rutea a:
    `user-service` (registro/login y JWT, H2), `product-service` (catálogo y stock en
    PostgreSQL) y `order-service` (reserva pedido validando stock en `product-service`).
    La discovery se hace con Eureka (`eureka-server`), y el flujo entre servicios usa
    load balancing por nombre de servicio (`lb://...` y `RestTemplate` @LoadBalanced).

P2. ¿Qué puertos usa cada módulo y por qué es importante mantenerlos alineados?

R2. Port convention:
    - Eureka: `8761`
    - API Gateway: `8080`
    - user-service: `8081`
    - product-service: `8082`
    - order-service: `8083`
    Es importante porque los endpoints demo/curl del README y la configuración de Eureka/Gateway
    dependen de que cada app escuche en su puerto esperado.

P3. ¿Qué versión de Java y de framework base se usa?

R3. Java `17` (en el pom raíz) y Spring Boot `3.2.0` (spring-boot-starter-parent).
    Spring Cloud se alinea con `2023.0.0`.

================================================================================
BLOQUE 2 — ARQUITECTURA Y FLUJO DE REQUESTS
================================================================================

P4. ¿Cómo llega una request “real” desde el cliente hasta el servicio correcto?

R4. El cliente llama a `http://localhost:8080` y el Gateway decide ruta con
    predicates por Path:
    - `/api/users/**` → `lb://user-service`
    - `/api/products/**` → `lb://product-service`
    - `/api/orders/**` → `lb://order-service`
    El `lb://...` se resuelve vía Eureka: si el servicio no está registrado, el Gateway
    puede devolver `503`.

P5. ¿Cómo se implementa el “cableado” entre order-service y product-service?

R5. `order-service` usa un `RestTemplate` con `@LoadBalanced` (en `RestClientConfig`).
    En `OrderService.reserve()` se llama:
      `http://product-service/api/products/stock/{productId}`
    Gracias a `@LoadBalanced`, `product-service` se resuelve por nombre (Eureka),
    no por IP/puerto fijo.

P6. Enumera endpoints principales y describe el payload de los que tienen body.

R6.
    - `user-service`:
      - `POST /api/users/register` body `RegisterRequest {name,email,password}`
      - `POST /api/users/login` body `LoginRequest {email,password}`
    - `product-service`:
      - `GET /api/products`
      - `GET /api/products/{id}`
      - `GET /api/products/stock/{productId}`
    - `order-service`:
      - `POST /api/orders/reserve` body `OrderReserveRequest {productId, quantity}`
        y respuesta `OrderReserveResponse {productId, requestedQuantity, availableStock, remainingStock}`

================================================================================
BLOQUE 3 — API GATEWAY + EUREKA
================================================================================

P7. ¿Cómo configura el API Gateway las rutas?

R7. En `api-gateway/src/main/resources/application.yml` se declaran rutas en `spring.cloud.gateway.routes`.
    Cada route tiene:
    - `id` (por ejemplo `user-service`)
    - `uri: lb://<service-name>`
    - `predicates: - Path=/api/<...>/**`

P8. ¿Qué rol cumple el “discovery locator” en el Gateway?

R8. Se activa:
    `spring.cloud.gateway.discovery.locator.enabled=true`
    y `lower-case-service-id=true`, lo cual facilita routing basado en el
    service-id cuando aplica (además de las rutas declaradas explícitamente).

P9. [TRAMPA] Si el Gateway devuelve `503 No servers available`, ¿qué causa típica habrías visto?

R9. Normalmente arrancar el Gateway antes de que los servicios terminen de registrarse en Eureka.
    La fix es reiniciar `api-gateway` o esperar ~20-60s para que Eureka registre todas las instancias.

P10. ¿Cómo está configurado Eureka y por qué no se registra a sí mismo?

R10. En `eureka-server/application.yml`:
      `register-with-eureka:false` y `fetch-registry:false`.
    Así el servidor actúa solo como “registry” y no como cliente de sí mismo.

BLOQUE 4 — SEGURIDAD EN user-service (JWT)
================================================================================

P11. ¿Qué endpoints están permitidos sin autenticación en `user-service`?

R11. En `SecurityConfig`:
    - `POST /api/users/register`
    - `POST /api/users/login`
    - `GET /api/users/health`
    - `permitAll` para `/h2-console/**`
    Cualquier otra ruta (si existiera) exigiría autenticación.

P12. Explica el filtro JWT: qué lee del request y qué hace si el token es válido.

R12. `JwtAuthFilter` inspecciona header `Authorization`.
    - Si no existe o no empieza con `Bearer `, deja pasar sin auth.
    - Si existe, extrae el JWT con `substring(7)`,
      obtiene `userEmail` via `jwtService.extractUsername(jwt)` (Claims subject),
      carga el `UserDetails` por email, valida el token (`jwtService.isTokenValid`),
      y si es válido setea `SecurityContextHolder` con `UsernamePasswordAuthenticationToken`.

P13. ¿Dónde se define el secreto y la expiración del JWT?

R13. En `user-service/application.yml`:
    - `jwt.secret` (base64) y `jwt.expiration` (milisegundos).
    `JwtService` decodifica el secret con BASE64 decode y firma con HS256.

P14. ¿Qué estrategia de hash se usa para almacenar la contraseña?

R14. `BCryptPasswordEncoder` (en `SecurityConfig`) y el password se guarda codificado
    en `AuthService.register()` con `passwordEncoder.encode(request.getPassword())`.

P15. [TRAMPA] ¿Los endpoints de productos u órdenes requieren JWT?

R15. No en este estado. La seguridad JWT está implementada en el `user-service`.
    `product-service` y `order-service` no declaran Spring Security en sus poms
    ni configs de filtros, así que sus endpoints son accesibles sin token.

BLOQUE 5 — product-service: modelo, persistencia y stock
================================================================================

P16. ¿Cómo está modelado `Product` y qué tipos usa?

R16. `Product` es una entidad JPA `@Entity` mapeada a `products` con:
    - `id: Long` (IDENTITY)
    - `name: String` (nullable=false)
    - `description: String`
    - `price: BigDecimal` (nullable=false)
    - `stock: Integer` (nullable=false)

P17. ¿Qué endpoints del catálogo expone `ProductController` y qué hace el path regex?

R17. Exponer:
    - `GET /api/products` → lista `ProductDto`
    - `GET /api/products/{id:[0-9]+}` → `getById`
    - `GET /api/products/stock/{productId}` → `stock`
    El regex fuerza que `id` sea numérico, evitando que paths no numéricos caigan en la
    ruta equivocada.

P18. ¿Cómo se inicializa la data si la BD está vacía?

R18. `ProductDataInitializer` es un `CommandLineRunner`.
    - Si `productRepository.count() > 0` retorna.
    - Si no, guarda 2 productos:
      - Keyboard: price 79.99, stock 25
      - Mouse: price 29.50, stock 40

P19. [TRAMPA] Cuando haces “reserve” en order-service, ¿el stock se decrementa persistido en product-service?

R19. No. `OrderService.reserve()` calcula `remainingStock` usando el stock consultado,
    pero no actualiza ni persiste el nuevo stock en PostgreSQL.
    Solo hay validación + respuesta calculada.

BLOQUE 6 — order-service: reglas de negocio y códigos HTTP
================================================================================

P20. Describe la lógica exacta de reserva (reserve) y el contrato de respuesta.

R20. En `OrderService.reserve()`:
    1) Llama a `product-service` con `GET /api/products/stock/{productId}` para obtener `ProductStockResponse`.
    2) Si el response es `null`, lanza `ResponseStatusException(404, "Product not found: ...")`.
       En la práctica, si el `product-service` no encuentra el producto, su `ProductService.getById()`
       lanza una `RuntimeException` (típicamente termina en `500`), y `order-service` podría ver un error
       del `RestTemplate` en vez de pasar por el caso `stock == null`.
    3) Si `stock.stock() < request.quantity()`, lanza `ResponseStatusException(409, "Insufficient stock ...")`.
    4) Si hay stock suficiente, calcula `remaining = stock.stock() - request.quantity()`
       y devuelve `OrderReserveResponse(productId, quantity, availableStock=stock.stock(), remainingStock=remaining)`.

P21. ¿Qué valida el sistema antes de intentar reservar?

R21. En este estado:
    - Para order-reserve no hay `jakarta.validation` ni validaciones anotadas en `OrderReserveRequest`.
    - Solo existe validación implícita de negocio por comparación con stock y por presencia del producto.
    Esto significa que no se bloquea, por ejemplo, `quantity <= 0` de forma explícita.

P22. ¿Qué pasa si el `productId` no existe?

R22. El código intenta lanzar `404` si `RestTemplate.getForObject(...)` devuelve `null`.
    Sin embargo, si el producto no existe, `productService.getStock()` acaba en `getById()` que lanza una `RuntimeException`
    (típicamente `500`), y entonces el comportamiento del cliente depende del `RestTemplate`
    (podría lanzar excepción en vez de devolver `null`).

P23. [TRAMPA] ¿Qué endpoint usa order-service para validar stock: “/products/{id}” o “/products/stock/{id}”?

R23. Usa específicamente `GET /api/products/stock/{productId}`.
    Ese endpoint devuelve un `StockResponse` compacto para evitar cargar todo el DTO del producto.

BLOQUE 7 — Integración técnica (RestTemplate y dependencias)
================================================================================

P24. ¿Por qué existe `RestClientConfig`?

R24. Porque el proyecto necesita un `RestTemplate` con `@LoadBalanced`.
    Sin eso, `RestTemplate` intentaría resolver la URL literal `http://product-service/...`
    sin pasar por Eureka.

P25. ¿Qué hace exactamente `@LoadBalanced` en este caso?

R25. Intercepta el `RestTemplate` para que cuando el host es un service-id,
    el request se enruta contra instancias registradas en Eureka (balanceador).

P26. ¿Cómo se traduce “falta de registro en Eureka” a fallos visibles en el cliente?

R26. En Gateway: normalmente `503 No servers available`.
    En orden-service: el `RestTemplate` puede fallar en la resolución o devolver error HTTP,
    según el timing y cómo maneje load balancer la falta de instancias.
